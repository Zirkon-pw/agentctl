# Структура файлов и артефактов

## Корневая служебная структура проекта

После инициализации Agent-CLI внутри репозитория создается каталог `.agentctl/`:

```text
.agentctl/
  config.yaml
  agents.yaml
  routing.yaml
  runtime/
  clarifications/
  guidelines/
  templates/
    tasks/
    prompts/
  tasks/
  runs/
  context/
  reviews/
```

Эта структура нужна для того, чтобы мета-логика агентной разработки жила рядом с кодом проекта и была воспроизводимой.

## Ожидаемая структура репозитория

Помимо `.agentctl/`, для самого CLI-проекта удобно придерживаться следующего корневого устройства:

```text
cmd/
internal/
tests/
.agentctl/
```

Здесь:

- `cmd/` содержит entrypoint бинарника;
- `internal/` содержит production-код CLI;
- `tests/` содержит все автоматические тесты, фикстуры и вспомогательные test utilities;
- `.agentctl/` содержит runtime, задачи, шаблоны и артефакты выполнения.

Отдельная директория `tests/` является проектным правилом: unit, integration, runtime и e2e тесты не должны размазываться по остальным каталогам.

## Назначение файлов и папок

### `.agentctl/config.yaml`

Базовая конфигурация проекта. Обычно здесь описываются общие правила исполнения, дефолтные настройки и базовые параметры среды.

### `.agentctl/agents.yaml`

Реестр доступных агентов. Файл описывает, какие исполнители доступны проекту и чем они отличаются друг от друга.

### `.agentctl/routing.yaml`

Правила маршрутизации задач между агентами. Содержит policy, по которой система выбирает исполнителя в зависимости от типа задачи, режима или других признаков.

### `.agentctl/runtime/`

Runtime-состояние активных задач. Здесь удобно хранить служебные данные для наблюдения и управления живыми run-процессами:

- список активных задач;
- pid или session id;
- heartbeat;
- runtime locks;
- сигналы `stop` и `kill`.

### `.agentctl/clarifications/`

Каталог отдельных `.yml`-уточнений по задачам. Здесь хранятся:

- clarification request files, созданные системой;
- clarification files, прикрепленные пользователем;
- история уточнений по каждой задаче.

### `.agentctl/guidelines/`

Каталог проектных правил. Обычно здесь лежат Markdown-документы вроде архитектурных ограничений, тестовой политики и миграционных правил.

### `.agentctl/templates/`

Общий каталог шаблонов. Его удобно делить на два подкаталога:

- `templates/tasks/` для task templates;
- `templates/prompts/` для prompt templates.

Task templates стандартизируют структуру задачи, а prompt templates стандартизируют сценарий поведения агента.

### `.agentctl/tasks/`

Спецификации задач в формате `.yml`. Здесь хранятся формализованные постановки задач, их scope, ограничения, привязка к агентам, подключенные prompt templates и ожидаемые проверки.

Validation-команды внутри задач и шаблонов должны учитывать принятую структуру репозитория и ссылаться на сценарии из `tests/`, если проект использует раздельные test suites.

### `.agentctl/runs/`

История конкретных запусков. Каждый run сохраняет артефакты выполнения и служит основой для review, сравнения и аудита.

### `.agentctl/context/`

Собранные context pack для задач. Здесь может храниться список включенных файлов, summaries и подготовленные материалы, которые были переданы агенту.

### `.agentctl/reviews/`

Результаты человеческой или агентной проверки. Каталог нужен для фиксации решений и замечаний после выполнения задачи.

## Как выглядит хранение результатов выполнения

README предлагает сохранять артефакты запуска в отдельной папке задачи, например:

```text
.agentctl/
  clarifications/
    TASK-001/
      clarification_request_001.yml
      clarification_001.yml
  runs/
    TASK-001/
      RUN-001/
        metadata.json
        session.json
        protocol.ndjson
        artifacts.json
        validation.json
        review_report.json
        stages/
          STAGE-001/
            stage_spec.json
            task_snapshot.yml
            prompt.md
            prompt-template.lock.yml
            adapter.stderr.log
            raw.stdout.log
            raw.stderr.log
            runtime_errors.log
```

## Что означает каждый артефакт

### `clarification_request_001.yml`

Файл запроса на уточнение, который система создает, если шаблон `clarify_if_needed` обнаруживает неоднозначность задачи.

### `clarification_001.yml`

Файл уточнения, который пользователь генерирует, заполняет и прикрепляет к задаче.

### `session.json`

Основной source of truth по session: статус, текущий агент, история stage, последнее состояние runtime и итоговые результаты stage-level пайплайна.

### `artifacts.json`

Нормализованный manifest всех артефактов session. Именно по нему команды `result show`, `result diff`, `result list` и `task logs` находят summary, diff, validation и диагностические логи.

### `prompt.md`

Финальный execution prompt, который был передан агенту.

### `prompt-template.lock.yml`

Снимок подключенных prompt templates и их параметров на момент запуска. Нужен для воспроизводимости и аудита поведения агента.

### `attached_clarifications.json`

Машиночитаемый список `.yml`-уточнений, которые были прикреплены к run или использованы при повторном `task run`.

### `summary.md`

Runtime-generated summary артефакт. Кратко описывает, что сделал агент, какие решения принял и какие риски видит.

### `diff.patch`

Runtime-generated patch с фактическими изменениями workspace. Это один из главных артефактов для review.

### `changed_files.json`

Runtime-generated машиночитаемый список workspace-файлов, затронутых выполнением.

### `protocol.ndjson`

Session-level сырой поток protocol events для `protocol_adapter` агентов.

### `adapter.stderr.log`

Stage-level stderr лог для protocol adapters.

### `raw.stdout.log` и `raw.stderr.log`

Stage-level логи для `raw_cli` агентов, которые не умеют machine-readable protocol.

### `runtime_errors.log`

Ошибки supervisor-а и driver-слоя: parser failures, unsupported stage, protocol contract violations и другие runtime-проблемы.

### `validation.json`

Результаты build, test, lint и других проверок, которые были запущены после работы агента. Включает историю повторных попыток (retry history) при использовании режима `full` validation.

### `metadata.json`

Служебная информация о запуске: timestamps, выбранный агент, параметры окружения, идентификаторы и другие технические метаданные.

## Почему файловая структура важна

Файловая организация превращает работу с агентами в инженерный процесс:

- задачу можно воспроизвести;
- результат можно проверить отдельно от чата;
- историю можно анализировать по run-артефактам;
- активные задачи можно наблюдать и администрировать;
- параллельные задачи не смешиваются друг с другом.
