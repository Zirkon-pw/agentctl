# Шаблоны промптов и сценарии работы агента

## Что такое шаблон промпта

Шаблон промпта - это готовый сценарий поведения агента, который можно прикрепить к задаче. Если `task` описывает, что нужно сделать, то шаблон промпта описывает, как агент должен себя вести во время выполнения.

Шаблон нужен, чтобы не повторять вручную одно и то же:

- задавать уточнения, если задача неоднозначна;
- сначала строить план, а потом выполнять;
- работать в строгом режиме без лишних предположений;
- выполнять только исследование;
- работать только как reviewer.

## Встроенные и пользовательские шаблоны

CLI должен поставляться с набором встроенных шаблонов. Это основной режим работы с шаблонами в первой версии.

Дополнительно пользователь может добавлять свои шаблоны из файлов.

### Встроенные шаблоны CLI

Встроенные шаблоны доступны сразу после установки:

- `clarify_if_needed`
- `plan_before_execution`
- `strict_executor`
- `research_only`
- `review_only`

### Пользовательские шаблоны

Пользовательские шаблоны подключаются из `.yml` файла и используются тогда, когда встроенного поведения недостаточно.

## Чем шаблон промпта отличается от шаблона задачи

### Шаблон задачи

Шаблон задачи стандартизирует структуру task spec:

- какие поля надо заполнить;
- какие guidelines подключить;
- какие команды validation задать по умолчанию.

### Шаблон промпта

Шаблон промпта стандартизирует поведение агента:

- задавать ли уточнения;
- нужен ли план до реализации;
- можно ли делать предположения;
- надо ли ограничиться только анализом;
- как должен выглядеть итоговый ответ.

Обе сущности используются вместе, но отвечают за разные части процесса.

## Как работать с шаблонами в CLI

### Посмотреть встроенные шаблоны

```bash
agentctl template list --builtin
agentctl template show clarify_if_needed
```

### Подключить встроенный шаблон при создании задачи

```bash
agentctl task create \
  --title "Implement refresh token repository" \
  --goal "Add repository for refresh tokens in auth module" \
  --agent claude \
  --template clarify_if_needed \
  --template strict_executor
```

### Дополнить существующую задачу встроенным шаблоном

```bash
agentctl task update TASK-001 --add-template clarify_if_needed
```

### Добавить пользовательский шаблон из файла

```bash
agentctl template add ./templates/prompts/custom_backend_guard.yml
agentctl template list
```

## Набор заготовленных шаблонов

Ниже описан минимальный набор встроенных шаблонов, который стоит считать обязательным для CLI.

### `clarify_if_needed`

Назначение:
Если в задаче есть неясности, агент не должен импровизировать. Вместо этого он создает запрос на уточнение задачи.

Как работает:
- агент анализирует `task`;
- если все ясно, продолжает выполнение;
- если есть блокирующие вопросы, формирует `clarification_request.yml`;
- задача переводится в статус `needs_clarification`.

Что дает:
- вопросы становятся частью task flow, а не отдельной ad-hoc командой;
- уточнения хранятся отдельными `.yml` файлами;
- пользователь может дополнить задачу и вернуть ее в работу.

### `plan_before_execution`

Назначение:
Перед изменением кода агент сначала строит план реализации.

Как работает:
- агент формирует план и impact analysis;
- задача может перейти в промежуточный статус ожидания подтверждения;
- после подтверждения или продолжения запускается исполнение.

### `strict_executor`

Назначение:
Агент работает максимально дисциплинированно, без лишних предположений и без выхода за scope.

Когда нужен:
- продовые изменения;
- рискованные изменения в backend;
- задачи с жесткими architecture rules.

### `research_only`

Назначение:
Агент не меняет код, а только анализирует систему и предлагает решение.

Когда нужен:
- архитектурное исследование;
- подготовка большой задачи;
- первичное изучение незнакомого модуля.

### `review_only`

Назначение:
Шаблон для независимого review без модификации кода.

Когда нужен:
- второй агент как reviewer;
- критичные изменения;
- приемка с независимой оценкой.

## Уточнения по задаче как шаблон, а не отдельная команда

Работа с вопросами больше не должна описываться как отдельная команда вида `task questions`.

Правильная модель такая:

1. К задаче подключается встроенный шаблон `clarify_if_needed`.
2. `task run` пытается выполнить задачу.
3. Если неясностей нет, задача идет дальше по обычному pipeline.
4. Если неясности есть, система создает `clarification_request.yml`.
5. Пользователь генерирует файл уточнения через утилиту `clarification`.
6. Заполненный `.yml` прикрепляется к задаче.
7. Задача переводится в статус `ready_to_resume`.
8. `task resume` возвращает задачу в выполнение.

То есть вопросы - это разновидность шаблонного поведения задачи, а не отдельный интерактивный режим.

## Формат пользовательского шаблона промпта

Пользовательский шаблон удобно хранить как `.yml` файл:

```yaml
id: custom_backend_guard
title: Backend guard
description: Enforce strict backend rules before any code changes.
kind: prompt_template

behavior:
  require_explicit_scope: true
  clarification_if_ambiguous: true
  allow_non_blocking_assumptions: false

output:
  execution_artifacts:
    - diff.patch
    - summary.md
    - validation.json

template: |
  Review the task, scope and guidelines before editing code.
  If anything is ambiguous, request clarification instead of making assumptions.
  Keep changes inside allowed paths and return the expected artifacts.
```

## Что важно предусмотреть в первой версии

Для первой версии достаточно следующего:

1. Встроенные шаблоны уже доступны в CLI.
2. Задача может быть создана сразу с набором встроенных шаблонов.
3. Шаблоны можно добавлять к уже существующей задаче.
4. `clarify_if_needed` создает `clarification_request.yml`, а не отдельный интерактивный вопросный режим.
5. Пользовательские шаблоны можно импортировать из файла, но основная работа строится на встроенных шаблонах.
