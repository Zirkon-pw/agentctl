# Конфигурационные файлы проекта

## Общая идея конфигурации

Agent-CLI предполагает несколько уровней конфигурации:

- общая конфигурация проекта;
- описание доступных агентов;
- routing policy;
- runtime policy;
- guidelines как нормативные документы;
- шаблоны задач;
- встроенные и пользовательские шаблоны промптов;
- clarification policy;
- task-спецификации.

Ниже описана рекомендуемая структура этих файлов на основе исходного README.

## `config.yaml`

`config.yaml` хранит базовые настройки проекта и исполнения. README не задает жесткую схему, но логично использовать этот файл для следующих групп параметров:

- идентичность проекта;
- дефолтный агент;
- режим изоляции выполнения;
- общие правила validation;
- дефолтное поведение prompt templates;
- список встроенных шаблонов CLI;
- правила хранения clarification files;
- настройки runtime-наблюдения и graceful shutdown;
- параметры хранения артефактов.

Пример:

```yaml
project:
  name: auth-service
  language: dotnet

execution:
  default_agent: codex
  isolation: worktree
  mode: strict

prompting:
  builtin_templates:
    - clarify_if_needed
    - plan_before_execution
    - strict_executor
    - research_only
    - review_only
  default_template: strict_executor
  allow_multiple_templates: true

clarifications:
  dir: .agentctl/clarifications
  strategy: by_yml_files
  allow_multiple_files: true

runtime:
  max_parallel_tasks: 4
  heartbeat_interval_sec: 5
  stale_after_sec: 30
  graceful_stop_timeout_sec: 20
  allow_force_kill: true

validation:
  default_commands:
    - dotnet build
    - dotnet test

artifacts:
  runs_dir: .agentctl/runs
  context_dir: .agentctl/context
  reviews_dir: .agentctl/reviews
```

## `agents.yaml`

`agents.yaml` описывает доступных исполнителей. README указывает, что у агента есть специализация, сильные стороны, стоимость, скорость, ограничение контекста, режимы работы и набор инструментов.

Пример:

```yaml
agents:
  - id: claude
    role: executor
    specialization:
      - architecture_refactor
      - deep_analysis
    strengths:
      - large_context_reasoning
      - architecture_review
    speed: medium
    cost: high
    context_limit: large
    modes:
      - strict
      - research
    tools:
      - filesystem
      - git

  - id: codex
    role: executor
    specialization:
      - code_generation
      - task_execution
    strengths:
      - code_edits
      - terminal_workflow
    speed: high
    cost: medium
    context_limit: medium
    modes:
      - strict
      - fast
    tools:
      - filesystem
      - terminal
```

## `routing.yaml`

`routing.yaml` определяет, как система выбирает исполнителя под конкретную задачу.

Пример:

```yaml
routing:
  - when: task.type == "architecture_refactor"
    agent: claude
  - when: task.type == "code_generation"
    agent: codex
  - when: task.type == "bulk_tests"
    agent: qwen
```

Такой файл нужен, когда в системе несколько агентов и выбор лучше формализовать, чем делать вручную каждый раз.

## `runtime` в `config.yaml`

Runtime-конфигурация определяет, как именно система наблюдает и останавливает живые задачи.

Особенно полезны следующие параметры:

- `max_parallel_tasks`
- `heartbeat_interval_sec`
- `stale_after_sec`
- `graceful_stop_timeout_sec`
- `allow_force_kill`

Эти настройки нужны, чтобы команды вроде `task ps`, `task stop` и `task kill` имели предсказуемое поведение.

## `guidelines/`

Guidelines не являются YAML-конфигом, но они критичны для поведения системы. Это набор Markdown-документов, которые описывают:

- архитектурные ограничения;
- кодстайл;
- правила тестирования;
- требования к миграциям;
- стандарты логирования;
- требования к безопасности.

Примеры файлов, прямо упомянутых в README:

- `backend-csharp-guidelines.md`
- `ddd-rules.md`
- `database-migrations-policy.md`
- `testing-policy.md`

## `templates/tasks/`

Каталог шаблонов задач хранит типовые описания задач. В README приведен пример шаблона `add-repository`:

```yaml
name: add-repository
fields:
  - module
  - entity
  - database
defaults:
  validation:
    - dotnet build
    - dotnet test
attached_guidelines:
  - backend-guidelines
  - ddd-rules
```

Назначение шаблонов:

- сократить ручную постановку задач;
- стандартизировать task spec;
- автоматически прикреплять правила;
- заранее предлагать проверки.

## `templates/prompts/`

Каталог пользовательских prompt templates хранит кастомные сценарии поведения агента. В отличие от task templates, эти шаблоны не описывают бизнес-содержание задачи, а описывают режим ее исполнения.

Пример:

```yaml
id: custom_backend_guard
behavior:
  clarification_if_ambiguous: true
  require_explicit_scope: true
template: |
  Before making code changes, analyze the task and ask only blocking
  questions. Do not edit files until clarification is attached.
```

Назначение prompt templates:

- стандартизировать поведение агента;
- подключать пользовательскую логику поверх встроенных шаблонов;
- управлять строгостью исполнения;
- разделять режимы executor, planner, researcher и reviewer.

## Встроенные шаблоны CLI

Основной сценарий первой версии - использовать встроенные шаблоны, которые уже поставляются вместе с CLI:

- `clarify_if_needed`
- `plan_before_execution`
- `strict_executor`
- `research_only`
- `review_only`

Они не требуют файлового импорта и подключаются к задаче по имени.

## `clarifications/`

Каталог `clarifications/` хранит отдельные `.yml`-уточнения по задачам.

В нем лежат:

- request files, созданные системой;
- clarification files, прикрепленные пользователем.

Именно эти файлы должны передаваться в задачу как отдельный слой контекста.

## `tasks/*.yml`

Отдельные файлы задач содержат уже конкретную постановку задачи для исполнения. Они описывают:

- цель;
- scope;
- контекст;
- подключенные prompt templates;
- ссылки на clarification files;
- task-specific runtime policy;
- ограничения;
- validation;
- выбранного агента;
- статус и историю выполнения.

Подробная структура task spec вынесена в отдельный документ: [YML-структура задачи](../задачи/структура_yml_задачи.md).
