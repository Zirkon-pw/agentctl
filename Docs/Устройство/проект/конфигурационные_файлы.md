# Конфигурационные файлы проекта

## Общая идея конфигурации

Agent-CLI предполагает несколько уровней конфигурации:

- общая конфигурация проекта;
- описание доступных агентов;
- routing policy;
- guidelines как нормативные документы;
- шаблоны задач;
- шаблоны промптов;
- task-спецификации.

Ниже описана рекомендуемая структура этих файлов на основе исходного README.

## `config.yaml`

`config.yaml` хранит базовые настройки проекта и исполнения. README не задает жесткую схему, но логично использовать этот файл для следующих групп параметров:

- идентичность проекта;
- дефолтный агент;
- режим изоляции выполнения;
- общие правила validation;
- дефолтное поведение prompt templates;
- параметры хранения артефактов.

Пример:

```yaml
project:
  name: auth-service
  language: dotnet

execution:
  default_agent: codex
  isolation: worktree
  mode: strict

prompting:
  default_template: strict-executor
  allow_multiple_templates: true
  clarification_answers_file: answers.yml

validation:
  default_commands:
    - dotnet build
    - dotnet test

artifacts:
  runs_dir: .agentctl/runs
  context_dir: .agentctl/context
  reviews_dir: .agentctl/reviews
```

## `agents.yaml`

`agents.yaml` описывает доступных исполнителей. README указывает, что у агента есть специализация, сильные стороны, стоимость, скорость, ограничение контекста, режимы работы и набор инструментов.

Пример:

```yaml
agents:
  - id: claude
    role: executor
    specialization:
      - architecture_refactor
      - deep_analysis
    strengths:
      - large_context_reasoning
      - architecture_review
    speed: medium
    cost: high
    context_limit: large
    modes:
      - strict
      - research
    tools:
      - filesystem
      - git

  - id: codex
    role: executor
    specialization:
      - code_generation
      - task_execution
    strengths:
      - code_edits
      - terminal_workflow
    speed: high
    cost: medium
    context_limit: medium
    modes:
      - strict
      - fast
    tools:
      - filesystem
      - terminal
```

## `routing.yaml`

`routing.yaml` определяет, как система выбирает исполнителя под конкретную задачу.

Пример:

```yaml
routing:
  - when: task.type == "architecture_refactor"
    agent: claude
  - when: task.type == "code_generation"
    agent: codex
  - when: task.type == "bulk_tests"
    agent: qwen
```

Такой файл нужен, когда в системе несколько агентов и выбор лучше формализовать, чем делать вручную каждый раз.

## `guidelines/`

Guidelines не являются YAML-конфигом, но они критичны для поведения системы. Это набор Markdown-документов, которые описывают:

- архитектурные ограничения;
- кодстайл;
- правила тестирования;
- требования к миграциям;
- стандарты логирования;
- требования к безопасности.

Примеры файлов, прямо упомянутых в README:

- `backend-csharp-guidelines.md`
- `ddd-rules.md`
- `database-migrations-policy.md`
- `testing-policy.md`

## `templates/tasks/`

Каталог шаблонов задач хранит типовые описания задач. В README приведен пример шаблона `add-repository`:

```yaml
name: add-repository
fields:
  - module
  - entity
  - database
defaults:
  validation:
    - dotnet build
    - dotnet test
attached_guidelines:
  - backend-guidelines
  - ddd-rules
```

Назначение шаблонов:

- сократить ручную постановку задач;
- стандартизировать task spec;
- автоматически прикреплять правила;
- заранее предлагать проверки.

## `templates/prompts/`

Каталог prompt templates хранит сценарии поведения агента. В отличие от task templates, эти шаблоны не описывают бизнес-содержание задачи, а описывают режим ее исполнения.

Пример:

```yaml
id: clarify-then-implement
mode: two_stage
behavior:
  clarification_required: true
  block_execution_until_answers: true
  max_questions: 5
template: |
  Before making code changes, analyze the task and ask only blocking
  questions. Do not edit files until answers are provided.
```

Назначение prompt templates:

- стандартизировать поведение агента;
- задавать двухэтапное выполнение;
- управлять строгостью исполнения;
- разделять режимы executor, planner, researcher и reviewer.

## `tasks/*.yml`

Отдельные файлы задач содержат уже конкретную постановку задачи для исполнения. Они описывают:

- цель;
- scope;
- контекст;
- подключенные prompt templates;
- ограничения;
- validation;
- выбранного агента;
- статус и историю выполнения.

Подробная структура task spec вынесена в отдельный документ: [YML-структура задачи](../задачи/структура_yml_задачи.md).
