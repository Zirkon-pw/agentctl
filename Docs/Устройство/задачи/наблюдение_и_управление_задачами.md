# Наблюдение и управление задачами

## Зачем нужен runtime control

После появления асинхронной очереди задач пользователь должен уметь не только ставить новую задачу, но и работать с уже существующими:

- видеть, какие задачи реально исполняются прямо сейчас;
- понимать, на каком шаге зависла задача;
- смотреть live-логи и события;
- мягко останавливать задачу;
- принудительно завершать зависший run;
- ставить задачу на паузу и потом возобновлять;
- повторно запускать неудачные или убитые задачи.

Без этого система остается удобной для запуска, но неудобной для эксплуатации.

## Какие задачи считаются существующими

Под существующими задачами здесь понимаются все задачи, которые уже есть в системе, независимо от их текущего состояния:

- `draft`
- `queued`
- `preparing_context`
- `running`
- `needs_clarification`
- `ready_to_resume`
- `paused`
- `stopping`
- `stopped`
- `killed`
- `validating`
- `review`
- `completed`
- `failed`
- `rejected`
- `canceled`

## Команды наблюдения

### Общий список задач

```bash
agentctl task list
```

Показывает реестр задач и их статусы. Это обзорная команда по всей системе.

### Псевдо `ps` по активным задачам

```bash
agentctl task ps
```

Команда нужна для наблюдения за живыми run-процессами. Она показывает только активные задачи и их runtime-состояние, например:

- `task_id`
- `run_id`
- `status`
- `agent`
- `worker`
- `pid` или `session_id`
- `started_at`
- `last_heartbeat`
- текущий шаг пайплайна

### Подробный inspect задачи

```bash
agentctl task inspect TASK-001
```

Эта команда дает расширенную карточку задачи:

- основную спецификацию;
- подключенные guidelines и шаблоны промптов;
- текущий статус;
- активный run;
- runtime metadata;
- attached clarification files;
- последние события;
- ссылки на артефакты.

### Просмотр логов

```bash
agentctl task logs TASK-001
agentctl task logs TASK-001 --follow
```

`logs` нужен для технической диагностики. Через него пользователь видит:

- системные сообщения оркестратора;
- stdout и stderr агента;
- сообщения validation;
- ошибки остановки или kill.

### Просмотр событий

```bash
agentctl task events TASK-001
agentctl task events TASK-001 --tail 50
```

События удобнее логов, когда нужен именно таймлайн изменений состояния:

- задача поставлена в очередь;
- собран контекст;
- отправлен prompt;
- создан `clarification_request.yml`;
- прикреплен `clarification_001.yml`;
- запрошен stop;
- run завершен;
- validation завершен.

### Наблюдение в реальном времени

```bash
agentctl task watch TASK-001
```

Эта команда объединяет статус, heartbeat, текущий шаг и последние события в один live-view.

### Наблюдение за уточнениями

```bash
agentctl clarification show TASK-001
```

Эта команда нужна, чтобы увидеть:

- есть ли pending clarification request;
- какие `.yml`-уточнения уже прикреплены;
- можно ли возвращать задачу в работу.

## Команды управления

### Мягкая остановка

```bash
agentctl task stop TASK-001
```

`stop` отправляет задаче graceful stop request. Система должна:

- не убивать процесс мгновенно;
- дать агенту завершить текущий безопасный шаг;
- сохранить логи, состояние и частичные артефакты;
- перевести задачу в `stopping`, а затем в `stopped`.

Это основной способ завершения живой задачи, если пользователь передумал или хочет сменить направление.

### Принудительное завершение

```bash
agentctl task kill TASK-001
```

`kill` нужен для зависших, зациклившихся или неотвечающих задач. Это force termination, поэтому после него:

- процесс завершается немедленно;
- часть артефактов может оказаться неполной;
- статус задачи переходит в `killed`;
- в runtime metadata фиксируется причина kill.

`kill` должен использоваться реже, чем `stop`.

### Пауза

```bash
agentctl task pause TASK-001
```

`pause` нужен, когда задачу надо временно заморозить без потери контекста. Для этого система должна поддерживать checkpoint boundary:

- задача доходит до безопасной точки;
- runtime сохраняет состояние;
- статус переходит в `paused`.

Если текущий исполнитель не поддерживает безопасную паузу в середине шага, pause может работать только между фазами пайплайна.

### Возобновление

```bash
agentctl task resume TASK-001
```

`resume` возвращает задачу из `paused`, `ready_to_resume` или `stopped` обратно в активное исполнение с тем же task context и текущим run либо с новым run, в зависимости от политики системы.

### Отмена до старта

```bash
agentctl task cancel TASK-001
```

`cancel` нужен для задач, которые еще не исполняются как живой процесс:

- `draft`
- `queued`
- `needs_clarification`

В этом случае не нужен stop или kill, потому что runtime-процесса еще нет или он не выполняет код.

### Повторный запуск

```bash
agentctl task rerun TASK-001
```

Повторный запуск нужен после:

- `failed`
- `stopped`
- `killed`
- отклонения на review

Новый run должен ссылаться на предыдущий, чтобы история оставалась прозрачной.

## Разница между `stop`, `kill` и `cancel`

### `cancel`

Используется до активного исполнения. Это отмена задачи в очереди или на стадии ожидания уточнения.

### `stop`

Используется для управляемой остановки активной задачи с сохранением максимума состояния.

### `kill`

Используется как аварийный рубильник, если задача зависла или runtime перестал отвечать.

## Какие данные должна показывать runtime-система

Чтобы `ps`, `inspect` и `watch` были полезными, runtime должен хранить:

- `task_id`
- `run_id`
- `agent`
- `worker_id`
- `pid` или `session_id`
- `status`
- `current_stage`
- `started_at`
- `updated_at`
- `last_heartbeat`
- `stop_requested_at`
- `kill_requested_at`
- `clarification_request_path`
- `attached_clarification_paths`
- `exit_reason`

## Когда этот функционал особенно важен

- при параллельном исполнении нескольких задач;
- при долгих задачах с большим контекстом;
- при очередях validation;
- при задачах, которые требуют `.yml`-уточнений;
- при зависаниях или таймаутах внешних инструментов.

## Минимальный набор для первой версии

Для первого рабочего runtime control достаточно поддержать:

1. `task ps`
2. `task inspect`
3. `task logs`
4. `task stop`
5. `task kill`
6. `task resume`
7. `task cancel`

`pause`, `watch` и подробный `events` могут идти следующим слоем, но модель лучше заложить сразу.
