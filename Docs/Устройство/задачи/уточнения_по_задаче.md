# YML уточнения по задаче

## Зачем нужны отдельные clarification files

Уточнения по задаче не должны растворяться в чате или в отдельных ad-hoc командах. Если задача требует прояснения, это должно оформляться отдельными `.yml` файлами, которые:

- существуют отдельно от основного `task.yml`;
- прикрепляются к задаче как дополнительные входные данные;
- участвуют в повторном запуске или `resume`;
- сохраняются как часть воспроизводимого процесса.

## Когда появляются уточнения

Обычно clarification files появляются, когда к задаче подключен встроенный шаблон `clarify_if_needed`.

Сценарий такой:

1. Пользователь запускает `task run`.
2. Агент анализирует задачу.
3. Если задача понятна, исполнение продолжается без уточнений.
4. Если задача неоднозначна, система создает `clarification_request_001.yml`.
5. Задача получает статус `needs_clarification`.
6. Пользователь генерирует рабочий `.yml` файл уточнения и прикрепляет его к задаче.
7. После этого задача получает статус `ready_to_resume`.

## Где хранятся файлы уточнений

Рекомендуемая структура:

```text
.agentctl/
  clarifications/
    TASK-001/
      clarification_request_001.yml
      clarification_001.yml
      clarification_002.yml
```

Здесь:

- `clarification_request_001.yml` - запрос на уточнение, созданный системой или агентом;
- `clarification_001.yml` - заполненный файл уточнения, который пользователь прикрепляет к задаче.

## Команды утилиты `clarification`

### Сгенерировать рабочий файл уточнения

```bash
agentctl clarification generate TASK-001
```

Команда создает editable `.yml` файл на основе pending clarification request.

### Посмотреть текущее состояние уточнений

```bash
agentctl clarification show TASK-001
```

Команда показывает:

- есть ли pending request;
- какие clarification files уже прикреплены;
- готова ли задача к `resume`.

### Прикрепить уточнение к задаче

```bash
agentctl clarification attach TASK-001 .agentctl/clarifications/TASK-001/clarification_001.yml
```

После прикрепления система:

- валидирует файл;
- добавляет его к задаче;
- обновляет блок `clarifications` в task spec;
- переводит задачу в `ready_to_resume`.

### Вернуть задачу в работу

```bash
agentctl task resume TASK-001
```

Уточнение не исполняет задачу само по себе. Оно только дополняет task context. Реальное продолжение работы происходит через `task resume`.

## Формат `clarification_request_001.yml`

Пример запроса на уточнение:

```yaml
task_id: TASK-001
request_id: CLAR-REQ-001
created_by: claude
reason: Missing business rules for refresh token rotation

questions:
  - id: q1
    text: Should refresh tokens be single use or reusable?
  - id: q2
    text: Do we need DB migration in this task or only repository logic?

context_refs:
  - src/Auth/Application/LoginHandler.cs
  - docs/backend-guidelines.md
```

Этот файл система создает сама. Пользователь его не редактирует.

## Формат `clarification_001.yml`

Пример файла, который пользователь генерирует и заполняет:

```yaml
task_id: TASK-001
request_id: CLAR-REQ-001
clarification_id: CLAR-001

answers:
  q1: Single use
  q2: Repository logic and migration in the same task

notes:
  - Keep public API unchanged
  - Add integration tests for persistence
```

Этот файл прикрепляется к задаче и становится частью ее контекста.

## Как clarification files влияют на задачу

После прикрепления `.yml`-уточнения task spec должен отражать это в отдельном блоке:

```yaml
status: ready_to_resume

clarifications:
  pending_request: .agentctl/clarifications/TASK-001/clarification_request_001.yml
  attached:
    - .agentctl/clarifications/TASK-001/clarification_001.yml
```

Таким образом задача остается формализованной и воспроизводимой.

## Что важно для первой версии

Для первой версии достаточно поддержать:

1. генерацию рабочего `.yml` уточнения;
2. прикрепление файла к задаче;
3. возврат задачи в работу через `task resume`;
4. хранение clarification files отдельно от основного task file.
