# Жизненный цикл задачи

## Что такое задача

Задача в Agent-CLI - это формализованная инженерная единица работы. Она описывает не только цель, но и контекст, ограничения, правила и критерии проверки.

Именно вокруг задачи строится вся остальная система:

- контекстная упаковка;
- встроенные и пользовательские шаблоны промптов;
- выбор агента;
- изолированное выполнение;
- runtime-наблюдение;
- runtime-управление;
- validation;
- review;
- история запусков.

## Создание задачи

Задача может появиться двумя способами:

- вручную через `agentctl task create`;
- на основе task template из `templates/tasks/`.

При создании важно зафиксировать:

- название и цель;
- выбранного агента;
- разрешенные и запрещенные пути;
- подключенные guidelines;
- подключенные шаблоны промптов;
- нужные файлы и паттерны для контекста;
- ограничения вроде `require_tests` или `no_breaking_changes`;
- команды validation.

## Подготовка к запуску

Перед исполнением задача проходит стадию подготовки:

1. Проверяется и нормализуется task spec.
2. Система подбирает или подтверждает агента.
3. Context Builder собирает пакет контекста.
4. Prompt Builder применяет подключенные шаблоны и формирует финальный execution prompt.
5. Если шаблон `clarify_if_needed` обнаруживает неясность, система создает `clarification_request.yml` и переводит задачу в статус `needs_clarification`.
6. После прикрепления `.yml`-уточнения задача получает статус `ready_to_resume`.
7. После `task resume` создается или продолжается изолированная среда выполнения.

## Выполнение задачи

На стадии run агент получает подготовленный пакет и работает в рамках заданного scope. Система при этом должна контролировать:

- что агент не выходит за allowed paths;
- что forbidden paths не затрагиваются;
- что результат сохраняется как артефакты;
- что несколько задач не конфликтуют друг с другом.

Если задача подключена к шаблону `clarify_if_needed`, run может распасться на две части:

1. стадия запроса уточнения, где агент создает `clarification_request.yml`;
2. стадия исполнения, где агент продолжает работу после прикрепления `.yml`-уточнения.

Во время выполнения задача должна оставаться наблюдаемой и управляемой:

- через `task ps` пользователь видит активные run;
- через `task inspect`, `task logs` и `task events` получает техническую картину происходящего;
- через `task stop`, `task pause`, `task resume`, `task cancel` и `task kill` управляет уже существующими задачами.
- через утилиту `clarification` пользователь создает и прикрепляет `.yml`-уточнения, если задача требует прояснения.

В README предлагается выполнять задачи в изолированной среде, например через:

- `git worktree`;
- временную папку;
- отдельную ветку;
- контейнер.

## Validation pipeline

После завершения работы агента система запускает проверки. Это могут быть:

- build;
- lint;
- format;
- unit tests;
- integration tests;
- security checks;
- custom policy checks.

Именно эта стадия переводит результат из "агент что-то сделал" в "результат технически проверен".

## Review и приемка

После validation задача попадает в review. На этом этапе человек или reviewer-агент оценивает:

- diff;
- summary;
- touched files;
- результаты проверок;
- риски и assumptions.

Далее возможны действия:

- принять задачу;
- отклонить;
- отправить на повторный запуск;
- перенаправить другому агенту;
- запросить доработку.

## Статусы задачи

Статусы задачи - это официальная модель состояний task в системе.

Базовый набор статусов:

- `draft`
- `queued`
- `preparing_context`
- `running`
- `needs_clarification`
- `ready_to_resume`
- `pausing`
- `paused`
- `stopping`
- `stopped`
- `killed`
- `validating`
- `review`
- `completed`
- `failed`
- `rejected`
- `canceled`

Эти статусы позволяют видеть задачу как управляемый pipeline, а не как одноразовый ответ модели.

Отдельно важно различать:

- `needs_clarification` как состояние, в котором система ждет прикрепленного `.yml`-уточнения;
- `ready_to_resume` как состояние, в котором уточнение уже прикреплено и задача готова вернуться в работу;
- `stopped` как результат мягкой остановки;
- `killed` как результат принудительного завершения;
- `paused` как временно замороженное, но потенциально продолжаемое исполнение.

Подробная operational-модель вынесена в документ [Наблюдение и управление задачами](./наблюдение_и_управление_задачами.md).

## Run history

У одной задачи может быть несколько запусков. История запусков важна, потому что она показывает:

- каким агентом запускали задачу;
- какие шаблоны промптов были подключены;
- какой контекст был выдан;
- какие `.yml`-уточнения прикреплялись;
- как менялось runtime-состояние;
- были ли stop, pause или kill;
- что изменилось между попытками;
- почему одна попытка была отклонена, а другая принята.

Это особенно полезно, когда одна и та же задача сначала выполняется одним агентом, а затем reroute-ится другому.

## Параллельное выполнение

Одно из ключевых преимуществ Agent-CLI - возможность запускать несколько задач одновременно. Пользователь может поставить очередь независимых задач, а система сама ведет статусы и сохраняет результаты по каждой из них отдельно.

Такой режим делает взаимодействие с агентами похожим на управление инженерной очередью работ, а не на последовательную переписку в одном окне чата.
