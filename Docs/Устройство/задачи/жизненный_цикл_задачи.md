# Жизненный цикл задачи

## Что такое задача

Задача в Agent-CLI - это формализованная инженерная единица работы. Она описывает не только цель, но и контекст, ограничения, правила и критерии проверки.

Именно вокруг задачи строится вся остальная система:

- контекстная упаковка;
- prompt templates;
- выбор агента;
- изолированное выполнение;
- validation;
- review;
- история запусков.

## Создание задачи

Задача может появиться двумя способами:

- вручную через `agentctl task create`;
- на основе task template из `templates/tasks/`.

При создании важно зафиксировать:

- название и цель;
- выбранного агента;
- разрешенные и запрещенные пути;
- подключенные guidelines;
- подключенные prompt templates;
- нужные файлы и паттерны для контекста;
- ограничения вроде `require_tests` или `no_breaking_changes`;
- команды validation.

## Подготовка к запуску

Перед исполнением задача проходит стадию подготовки:

1. Проверяется и нормализуется task spec.
2. Система подбирает или подтверждает агента.
3. Context Builder собирает пакет контекста.
4. Prompt Builder применяет prompt templates и формирует финальный execution prompt.
5. Если шаблон требует clarification stage, агент сначала подготавливает вопросы.
6. После ответов пользователя создается изолированная среда выполнения.

## Выполнение задачи

На стадии run агент получает подготовленный пакет и работает в рамках заданного scope. Система при этом должна контролировать:

- что агент не выходит за allowed paths;
- что forbidden paths не затрагиваются;
- что результат сохраняется как артефакты;
- что несколько задач не конфликтуют друг с другом.

Если задача подключена к двухэтапному prompt template, run разбивается на две части:

1. clarification stage, где агент задает вопросы;
2. execution stage, где агент уже меняет код после получения ответов.

В README предлагается выполнять задачи в изолированной среде, например через:

- `git worktree`;
- временную папку;
- отдельную ветку;
- контейнер.

## Validation pipeline

После завершения работы агента система запускает проверки. Это могут быть:

- build;
- lint;
- format;
- unit tests;
- integration tests;
- security checks;
- custom policy checks.

Именно эта стадия переводит результат из "агент что-то сделал" в "результат технически проверен".

## Review и приемка

После validation задача попадает в review. На этом этапе человек или reviewer-агент оценивает:

- diff;
- summary;
- touched files;
- результаты проверок;
- риски и assumptions.

Далее возможны действия:

- принять задачу;
- отклонить;
- отправить на повторный запуск;
- перенаправить другому агенту;
- запросить доработку.

## Статусы задачи

README предлагает следующую модель статусов:

- `draft`
- `queued`
- `preparing_context`
- `clarifying`
- `awaiting_answers`
- `running`
- `validating`
- `review`
- `completed`
- `failed`
- `rejected`

Эти статусы позволяют видеть задачу как управляемый pipeline, а не как одноразовый ответ модели.

## Run history

У одной задачи может быть несколько запусков. История запусков важна, потому что она показывает:

- каким агентом запускали задачу;
- какие prompt templates были подключены;
- какой контекст был выдан;
- что изменилось между попытками;
- почему одна попытка была отклонена, а другая принята.

Это особенно полезно, когда одна и та же задача сначала выполняется одним агентом, а затем reroute-ится другому.

## Параллельное выполнение

Одно из ключевых преимуществ Agent-CLI - возможность запускать несколько задач одновременно. Пользователь может поставить очередь независимых задач, а система сама ведет статусы и сохраняет результаты по каждой из них отдельно.

Такой режим делает взаимодействие с агентами похожим на управление инженерной очередью работ, а не на последовательную переписку в одном окне чата.
