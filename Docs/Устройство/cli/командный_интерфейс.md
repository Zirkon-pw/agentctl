# CLI и основные команды

## Роль CLI в продукте

`agentctl` - основной интерфейс управления системой. Через него пользователь инициализирует проект, подключает правила и шаблоны промптов, создает задачи, запускает агентов, работает с `.yml`-уточнениями, смотрит результаты и принимает решения по review.

CLI важен не только как оболочка для команд. Он задает рабочую модель продукта:

- задача оформляется как команда и файл спецификации;
- сценарий поведения агента оформляется как подключаемый встроенный или пользовательский шаблон промпта;
- выполнение запускается явно;
- результаты доступны через просмотр артефактов;
- существующие задачи можно наблюдать и управлять ими как runtime-процессами;
- структура CLI должна быть самообъяснимой через встроенную справку;
- статусы и история читаются как часть инженерного процесса;
- уточнения по задаче оформляются отдельными `.yml` файлами, а не отдельным question mode.

## Базовые группы команд

### Справка по CLI

```bash
agentctl help
agentctl help task
agentctl help task run
agentctl task run --help
```

`help` нужен как базовая точка входа в CLI, когда пользователь:

- только знакомится с системой;
- не помнит синтаксис команды;
- хочет быстро увидеть доступные подкоманды;
- проверяет флаги и примеры перед запуском.

Хорошая модель справки должна работать на трех уровнях:

- общая справка по `agentctl`;
- справка по командной группе, например `task` или `result`;
- справка по конкретной команде, например `task run`.

По сути, `help` должен быть встроенной картой CLI, а не формальной страницей с флагами.

### Инициализация проекта

```bash
agentctl init
```

Команда создает служебную структуру `.agentctl/`, в которой будут жить конфигурации, правила, шаблоны, задачи и результаты запусков.

### Подключение guidelines

```bash
agentctl guidelines add ./docs/backend-guidelines.md
agentctl guidelines add ./docs/ddd-rules.md
agentctl guidelines add ./docs/testing-policy.md
```

Guidelines добавляются в проект как нормативные документы, которые затем можно прикреплять к задачам.

### Встроенные шаблоны CLI

```bash
agentctl template list --builtin
agentctl template show clarify_if_needed
```

Встроенные шаблоны - это заготовленные сценарии поведения агента, которые уже поставляются вместе с CLI. Именно через них задача получает дополнительную логику выполнения.

Минимальный набор встроенных шаблонов:

- `clarify_if_needed`
- `plan_before_execution`
- `strict_executor`
- `research_only`
- `review_only`

### Добавление пользовательского шаблона

```bash
agentctl template add ./templates/prompts/custom_backend_guard.yml
agentctl template list
```

Пользовательские шаблоны нужны, когда встроенного набора недостаточно.

### Создание задачи

```bash
agentctl task create \
  --title "Implement refresh token repository" \
  --goal "Add repository for refresh tokens in auth module" \
  --agent claude \
  --template clarify_if_needed \
  --allow "src/Auth/**" \
  --forbid "src/Billing/**" \
  --guide backend-guidelines \
  --guide ddd-rules
```

На этом этапе пользователь описывает цель задачи, выбирает агента, задает scope, прикрепляет правила проекта и сценарий поведения агента.

### Добавление шаблона к существующей задаче

```bash
agentctl task update TASK-001 --add-template clarify_if_needed
```

Это позволяет дополнять задачу заготовленным шаблоном промпта без пересоздания task spec. У одной задачи может быть несколько шаблонов, если они совместимы по сценарию.

### Запуск задачи

```bash
agentctl task run TASK-001
```

Во время запуска система:

- собирает контекст;
- готовит execution prompt;
- применяет подключенные шаблоны;
- поднимает изолированную среду;
- передает задачу агенту;
- сохраняет артефакты;
- запускает validation.

Если к задаче подключен шаблон `clarify_if_needed`, `task run` не обязан немедленно завершаться кодовыми изменениями. Если агент видит блокирующие неясности, система создает `clarification_request.yml` и переводит задачу в статус `needs_clarification`.

### Работа с `.yml`-уточнениями

```bash
agentctl task run TASK-001
agentctl clarification generate TASK-001
agentctl clarification attach TASK-001 .agentctl/clarifications/TASK-001/clarification_001.yml
agentctl task resume TASK-001
```

Такой сценарий выглядит так:

1. `task run` запускает задачу с шаблоном `clarify_if_needed`.
2. Если уточнения не нужны, задача идет дальше по обычному pipeline.
3. Если уточнения нужны, система создает `clarification_request.yml`.
4. Утилита `clarification generate` создает рабочий `.yml` для заполнения.
5. Пользователь заполняет его и прикрепляет через `clarification attach`.
6. Задача получает статус `ready_to_resume`.
7. `task resume` возвращает ее в выполнение.

### Просмотр состояния очереди

```bash
agentctl task list
```

Команда показывает текущие статусы задач, например `draft`, `queued`, `preparing_context`, `running`, `needs_clarification`, `ready_to_resume`, `paused`, `stopping`, `stopped`, `killed`, `validating`, `review`, `completed`, `failed`, `rejected` или `canceled`.

### Наблюдение за активными задачами

```bash
agentctl task ps
agentctl task inspect TASK-001
agentctl task logs TASK-001 --follow
agentctl task events TASK-001 --tail 50
agentctl task watch TASK-001
```

Этот набор команд нужен для работы с уже существующими задачами и живыми run-процессами.

`task ps` показывает только активные задачи, похожим образом на системный `ps`.

`task inspect` раскрывает полное runtime-состояние конкретной задачи: текущий run, выбранного агента, heartbeat, текущую фазу и ссылки на артефакты.

`task logs` и `task events` используются для диагностики и анализа хода исполнения.

`task watch` дает live-view по задаче в реальном времени.

Подробная модель runtime-команд описана в документе [Наблюдение и управление задачами](../задачи/наблюдение_и_управление_задачами.md).

### Утилита уточнений

```bash
agentctl clarification generate TASK-001
agentctl clarification show TASK-001
agentctl clarification attach TASK-001 .agentctl/clarifications/TASK-001/clarification_001.yml
```

Утилита `clarification` нужна для работы с `.yml`-уточнениями по задаче.

`generate` создает рабочий файл уточнения на основе pending clarification request.

`show` показывает, какие уточнения уже запрошены или прикреплены.

`attach` прикрепляет заполненный `.yml` к задаче, чтобы она могла быть возвращена в работу.

Подробное описание clarification files вынесено в документ [YML уточнения по задаче](../задачи/уточнения_по_задаче.md).

### Просмотр результатов

```bash
agentctl result show TASK-001
agentctl result diff TASK-001
```

Через эти команды пользователь получает summary, список измененных файлов, patch, лог выполнения, результаты validation и отмеченные риски.

### Решения по review

```bash
agentctl task accept TASK-001
agentctl task reject TASK-001 --reason "Touched forbidden paths"
agentctl task rerun TASK-001
agentctl task route TASK-001 --agent codex
```

CLI поддерживает не только запуск, но и управление дальнейшей судьбой задачи: принятие, отклонение, повторное выполнение и перенаправление другому агенту.

### Управление существующими задачами

```bash
agentctl task stop TASK-001
agentctl task kill TASK-001
agentctl task pause TASK-001
agentctl task resume TASK-001
agentctl task cancel TASK-001
```

Команды управления нужны, когда задача уже создана и находится в runtime-системе.

`stop` делает мягкую остановку и старается сохранить состояние.

`kill` завершает run принудительно и нужен только для зависших задач.

`pause` и `resume` позволяют временно приостановить задачу, если runtime и агент поддерживают checkpoint boundary.

`cancel` используется для задач, которые еще не должны исполняться дальше, например в `queued` или `needs_clarification`.

## Практический сценарий через CLI

Типовой поток работы через консоль выглядит так:

1. При необходимости открыть справку через `agentctl help` или `agentctl help task run`.
2. Инициализировать агентную среду в репозитории.
3. Подключить guidelines, task templates и встроенные или пользовательские шаблоны промптов.
4. Создать задачу с ограничениями, агентом и нужными шаблонами.
5. Запустить задачу.
6. Если шаблон `clarify_if_needed` запросил уточнение, сгенерировать `.yml` через `clarification generate`, прикрепить его и вернуть задачу в работу через `task resume`.
7. Дождаться completion или review status.
8. При необходимости наблюдать задачу через `ps`, `inspect`, `logs` и `watch`.
9. Если run завис или потерял актуальность, использовать `stop`, `pause`, `resume`, `cancel` или `kill`.
10. Просмотреть diff и validation.
11. Принять, отклонить или повторно маршрутизировать результат.

## Почему CLI удобен именно для этого продукта

Для Agent-CLI консольный формат естественен, потому что он хорошо сочетается с:

- git и worktree;
- файлами `.yml` и `.md`;
- bash-командами для validation;
- автоматизацией и CI/CD;
- параллельным запуском независимых задач.
