# Как задача ставится агенту

## Общая модель взаимодействия

В Agent-CLI задача не отправляется агенту как произвольный текст. Перед исполнением она проходит несколько этапов нормализации, чтобы агент получил четкий контракт на работу.

Идея состоит в том, что пользователь задает намерение и ограничения, а система преобразует их в структурированный execution package.

## Что задает пользователь

При постановке задачи пользователь определяет:

- что нужно сделать;
- где можно вносить изменения;
- какие части проекта трогать нельзя;
- какие guidelines обязательны;
- какие prompt templates задают сценарий поведения агента;
- какого агента использовать;
- какие проверки нужно запустить после выполнения.

Это можно сделать через CLI-параметры или через файл спецификации задачи.

## Как система превращает задачу в пакет исполнения

### Формализация task spec

После создания задачи система сохраняет структурированную спецификацию с полями вроде `title`, `goal`, `scope`, `guidelines`, `prompt_templates`, `context`, `constraints` и `validation`.

### Построение scope

Scope задает границы поведения агента:

- `allowed_paths` показывают, где можно менять код;
- `forbidden_paths` явно запрещают лишние зоны;
- `must_read` фиксирует файлы, которые нужно изучить до начала работы.

### Сбор контекста

Context Builder не ограничивается только теми файлами, которые пользователь указал вручную. Он может добавить:

- связанные интерфейсы;
- ближайшие реализации;
- соседние тесты;
- summaries зависимых сущностей;
- релевантные конфиги и документацию.

Это дает баланс между нехваткой и переизбытком контекста.

### Применение prompt templates

Перед построением итогового prompt система подключает шаблоны поведения агента. Именно они определяют, должен ли агент:

- сразу переходить к реализации;
- сначала задать уточняющие вопросы;
- сначала построить план;
- работать только как reviewer;
- избегать любых допущений без подтверждения.

### Построение execution prompt

Prompt Builder собирает единый контракт для агента. Он нормализует задачу, применяет prompt templates и исключает двусмысленность, которая типична для чатов.

Пример итоговой структуры запроса:

```md
# Task
Implement refresh token repository in auth module.

## Goal
Add persistence for refresh tokens without breaking current auth flow.

## Allowed Paths
- src/Auth/Application/**
- src/Auth/Infrastructure/**

## Forbidden Paths
- src/Payments/**
- frontend/**

## Must Read
- docs/backend-guidelines.md
- src/Auth/Application/LoginHandler.cs
- src/Auth/Application/Interfaces/IRefreshTokenRepository.cs
- src/Auth/Infrastructure/DependencyInjection.cs

## Constraints
- Follow DDD boundaries
- Keep public API backward compatible
- Add tests for persistence logic
- Do not introduce breaking schema changes without migration

## Expected Output
- code changes
- summary of implementation
- list of modified files
- test results
- any risks or assumptions
```

## Что получает агент на вход

На этапе исполнения агенту передается не только промпт, но и связанный с ним рабочий пакет:

- snapshot задачи;
- подключенные prompt templates;
- context pack;
- список разрешенных и запрещенных путей;
- правила проекта;
- параметры окружения;
- expected output;
- validation criteria.

## Двухэтапное выполнение через шаблон вопросов

Если к задаче подключен шаблон типа `clarify-then-implement`, агент не начинает работу сразу. Вместо этого первый этап выглядит так:

1. Агент анализирует постановку задачи и ограничения.
2. Формирует только блокирующие вопросы.
3. Останавливается и ждет ответов пользователя.

После того как пользователь отвечает через CLI, система включает ответы в контекст и запускает второй этап, на котором агент уже реализует задачу.

Такая схема полезна в случаях, где опасно позволять агенту заполнять пробелы собственными предположениями.

## Что агент должен вернуть

Результат работы агента должен быть пригоден для review и повторного анализа. Поэтому система ожидает:

- кодовые изменения;
- summary реализации;
- список затронутых файлов;
- логи выполнения;
- результаты проверок;
- риски, допущения и замечания.

## Почему это лучше чата

В чате агент легко начинает додумывать за пользователя, теряет границы и меняет лишнее. В Agent-CLI задача ставится как инженерный контракт:

- цель сформулирована явно;
- ограничения проверяемы;
- контекст контролируется;
- сценарий поведения агента задается шаблоном, а не случайной фразой в чате;
- результат сохраняется как набор артефактов.
