# Маршрутизация, reviewer и режимы работы

## Выбор агента

В системе агент выбирается либо напрямую пользователем, либо через routing policy. На раннем этапе это может быть явный выбор:

```bash
agentctl task create --agent claude
agentctl task create --agent codex
agentctl task create --agent qwen
```

Такой подход удобен, когда пользователь сам знает сильные стороны каждого исполнителя.

## Routing policy

Когда задач становится больше, выбор агента можно автоматизировать через правила маршрутизации.

Пример структуры:

```yaml
routing:
  - when: task.type == "architecture_refactor"
    agent: claude
  - when: task.type == "code_generation"
    agent: codex
  - when: task.type == "bulk_tests"
    agent: qwen
```

Идея routing policy в том, чтобы подбирать агента по типу задачи, а не отправлять все на одну модель.

## Роли агентов

README описывает агента как логическую сущность со специализацией. Для практической работы это означает, что у агента могут быть разные роли:

- исполнитель кода;
- архитектурный аналитик;
- reviewer;
- дешевый исполнитель для массовых рутинных операций;
- локальный агент для чувствительных или бюджетных задач.

Эти роли удобно закреплять не только через routing, но и через prompt templates. Например один и тот же агент может запускаться как `strict-executor` в одном task и как `reviewer` в другом.

## Reviewer mode

Система может разделять роли исполнения и проверки. В этом режиме один агент реализует задачу, а второй получает уже готовые артефакты для независимого review.

Пример запуска:

```bash
agentctl task run TASK-004 --reviewer codex
```

После выполнения основного агента reviewer получает:

- diff;
- summary;
- validation report;
- список измененных файлов;
- контекст решения;
- reviewer-specific prompt template, если он задан.

Reviewer возвращает:

- замечания по качеству;
- риски;
- нарушения guidelines;
- рекомендации по доработке.

Такое разделение повышает надежность результата и снижает риск самооправдания со стороны одного и того же исполнителя.

## Режимы работы

README закладывает несколько режимов исполнения.

### Строгий режим

Подходит для продовых изменений. Предполагает:

- жесткий scope;
- запрет на лишние файлы;
- обязательный validation;
- обязательный review.

### Быстрый режим

Подходит для черновых и недорогих задач. Предполагает:

- меньше ограничений;
- более короткий pipeline;
- упрощенные проверки.

### Исследовательский режим

Подходит для анализа и планирования. В этом режиме агент может не менять код, а только:

- исследовать архитектуру;
- предлагать варианты решения;
- строить план декомпозиции;
- оценивать риски.

## Метрики и эволюция маршрутизации

В долгосрочной модели система может собирать статистику:

- какой агент лучше решает определенные типы задач;
- сколько retries требуется;
- как часто затрагиваются forbidden paths;
- каково среднее качество по validation;
- сколько времени и денег стоит исполнение.

Эти метрики позволяют делать routing policy не только ручной, но и основанной на реальных данных.
