# Системная модель и ключевые сущности

## Принципы, на которых строится система

### Не чат, а задача

Главная сущность продукта - `task`. Работа строится вокруг постановки задачи, а не вокруг свободного диалога с моделью.

### Контекст не должен быть избыточным

Агент получает не весь репозиторий, а только релевантные правила, файлы, summaries и зависимости, относящиеся к задаче.

### Агент - исполнитель, а не магический собеседник

Агент получает структурированный контракт на выполнение, работает в рамках ограничений и возвращает проверяемый результат.

### Результат должен быть артефактом

Итог работы фиксируется как набор файлов и отчетов, а не растворяется в истории переписки.

### Выполнение должно быть воспроизводимым

Система должна сохранять постановку задачи, выданный контекст, выбранного агента, проверки и итоговые артефакты.

### Параллельность должна быть управляемой

Несколько задач могут выполняться одновременно, но каждая должна быть изолирована и наблюдаема отдельно.

## Основные сущности

### Проект

Проект - это рабочее пространство, в котором лежат:

- конфигурация системы;
- набор агентов;
- routing-политики;
- guidelines;
- шаблоны задач;
- шаблоны промптов;
- сами задачи;
- история запусков и результаты;
- review-артефакты.

### Guidelines

Guidelines - это нормативный слой проекта. Они задают правила, которым агент обязан следовать:

- кодстайл;
- архитектурные ограничения;
- правила именования;
- политику тестирования;
- требования к безопасности;
- миграционные правила;
- definition of done.

### Task Templates

Шаблоны задач позволяют стандартизировать постановку типовых работ. Они определяют:

- обязательные поля;
- значения по умолчанию;
- рекомендуемые validation-команды;
- автоматически подключаемые guidelines.

### Prompt Templates

Шаблоны промптов задают сценарий поведения агента поверх самой задачи. Они отвечают не на вопрос "что сделать", а на вопрос "как именно вести себя при выполнении".

Система должна поддерживать два типа шаблонов:

- встроенные шаблоны CLI;
- пользовательские шаблоны из файла.

Через шаблон промпта можно зафиксировать:

- необходимость запросить уточнение по задаче;
- обязательное построение плана до начала правок;
- строгий режим без предположений;
- reviewer-only сценарий без внесения изменений.

### Task

Task - это формализованная задача, содержащая:

- идентификатор;
- название;
- цель;
- scope;
- ограничения;
- контекст;
- выбранного агента;
- подключенные шаблоны промптов;
- правила выполнения;
- критерии проверки;
- статус;
- итоговые результаты.

### Clarification File

Clarification file - это отдельный `.yml` файл, который уточняет постановку задачи и прикрепляется к ней после создания.

Он нужен, когда задача была остановлена не из-за ошибки, а из-за нехватки формализованной информации.

### Context Pack

Context Pack - это подготовленный пакет контекста для конкретной задачи. В него входят:

- текст задачи;
- связанные guidelines;
- обязательные и дополнительные файлы;
- summaries зависимых частей;
- ограничения по путям;
- validation rules.

### Agent

Agent - логическая сущность-исполнитель, а не просто запись о модели. Агент описывается через:

- специализацию;
- сильные стороны;
- стоимость;
- скорость;
- ограничения по контексту;
- режимы работы;
- доступные инструменты.

### Run

Run - это конкретное выполнение одной задачи. У одной задачи может быть несколько запусков с разными агентами, разным контекстом или после доработок.

Каждый run хранит:

- snapshot промпта;
- snapshot подключенных prompt templates;
- snapshot контекста;
- параметры окружения;
- output агента;
- diff;
- логи выполнения;
- результаты validation;
- временные метки.

### Runtime State

Runtime State описывает живое состояние уже существующей задачи или конкретного run во время исполнения.

Для runtime state важны:

- текущий статус;
- активный шаг пайплайна;
- worker;
- pid или session id;
- heartbeat;
- сигналы `pause`, `stop`, `kill`;
- причина завершения.

### Result / Artifact

Результат задачи сохраняется как артефакты:

- patch;
- список измененных файлов;
- summary;
- stdout и stderr;
- validation report;
- reviewer notes.

### Review Decision

После проверки задача получает одно из решений:

- `accepted`;
- `rejected`;
- `needs_changes`;
- `reroute`;
- `retry`.

## Сквозной поток работы

Типовой путь задачи в системе выглядит так:

1. Пользователь создает задачу вручную или по шаблону.
2. Система применяет guidelines, scope, ограничения и prompt templates.
3. Context Builder собирает контекстный пакет.
4. Prompt Builder строит нормализованный execution prompt с учетом шаблонов поведения агента.
5. Если встроенный шаблон `clarify_if_needed` обнаруживает неясность, система создает `clarification_request.yml` и переводит задачу в `needs_clarification`.
6. Пользователь генерирует и прикрепляет `.yml`-уточнение к задаче.
7. После этого задача получает статус `ready_to_resume` и может быть возвращена в исполнение.
8. Во время исполнения runtime layer публикует статус, heartbeat и события, а пользователь может выполнять `ps`, `inspect`, `logs`, `stop` или `kill`.
9. После выполнения запускается validation pipeline.
10. Результат сохраняется как артефакты.
11. Человек или reviewer-агент принимает итоговое решение.
