# Agent-CLI

---

# Общая идея

Это приложение для **агентной разработки**, в котором основной единицей взаимодействия является не чат, а **задача**.

Пользователь не “разговаривает с ИИ”, а **ставит инженерную задачу**:

* что нужно сделать,
* где это нужно сделать,
* какие правила нужно соблюдать,
* какие файлы и части проекта важны,
* что запрещено трогать,
* какой агент должен выполнять задачу,
* как проверить результат.

После этого система сама:

* собирает релевантный контекст,
* упаковывает его в рабочий пакет,
* отправляет задачу нужному агенту,
* сохраняет результат как артефакты,
* запускает проверки,
* показывает итог пользователю,
* дает принять, отклонить или отправить на доработку.

То есть это не AI chat wrapper, а **операционная среда для управляемой работы ИИ-агентов над кодом**.

Условно говоря, это смесь:

* task manager,
* orchestration engine,
* context packer,
* agent runtime,
* review system,
* dev workflow tool.

Если совсем коротко:
**это диспетчерская система, в которой нейроагенты работают как исполнители задач, а человек выступает как архитектор и принимающий инженер.**

---

# Какую проблему решает продукт
Главная проблема большинства существующих агентных решений в том, что они работают в формате чатов. А чат для разработки очень быстро превращается в туман.

В чате плохо фиксируется:

* точная постановка задачи,
* границы изменений,
* выданный контекст,
* набор ограничений,
* критерии готовности,
* воспроизводимость результата,
* причина, по которой агент принял то или иное решение.

Из-за этого начинается классическая картина:

* агенту дали слишком широкую задачу,
* контекста не хватило,
* он полез не туда,
* изменил лишнее,
* где-то недопонял архитектуру,
* а потом еще и сложно понять, на основании чего он вообще это сделал.

Проблема не только в самих моделях. Проблема в форме взаимодействия с ними.

Поэтому идея продукта такая:

**убрать диалоговую модель взаимодействия и заменить ее на task-based подход**.

Вместо:

> открыл чат, написал агенту, ждешь, уточняешь, передоговариваешься, скроллишь вверх, пытаешься вспомнить, что именно просил

Будет:

> создал задачу, указал scope, прикрепил правила, выбрал агента, запустил, получил diff и отчет, принял решение

Это сильно повышает:

* управляемость,
* качество,
* воспроизводимость,
* масштабируемость,
* прозрачность,
* возможность параллельной работы.

# Философия продукта

У продукта есть несколько базовых принципов.

## 1. Не чат, а задача

Основная сущность системы это task.

Не сообщение. Не беседа. Не сессия.
Именно формализованная задача.

## 2. Контекст не льется ведром

Контекст должен быть точечным и релевантным.

Агенту не нужно давать весь репозиторий. Ему нужно давать только то, что относится к задаче:

* нужные файлы,
* связанные интерфейсы,
* соседние тесты,
* правила проекта,
* краткие summaries зависимых частей.

## 3. Агент это исполнитель, а не “магический собеседник”

У агента есть роль.
Он получает структурированную задачу, работает в ограниченном контуре и возвращает проверяемый результат.

## 4. Результат должен быть артефактом

Результат работы агента должен сохраняться как:

* diff,
* patch,
* список измененных файлов,
* summary,
* лог выполнения,
* результаты тестов,
* замечания проверок.

Итог не должен растворяться в истории сообщений.

## 5. Все должно быть воспроизводимо

Нужно иметь возможность понять:

* какая была задача,
* какой контекст был выдан,
* какая модель использовалась,
* какие проверки запускались,
* какой именно результат был получен.

## 6. Параллельность должна быть управляемой

Можно выполнять много задач одновременно, но каждая должна быть:

* изолирована,
* отслеживаема,
* привязана к конкретному окружению,
* независима от других.

# Для кого этот продукт

В первую очередь это инструмент для:

* solo-разработчиков, которые хотят использовать несколько агентов осмысленно;
* small teams, где ИИ становится частью инженерного процесса;
* AI-first engineering setups;
* архитекторов и техлидов, которые хотят управлять агентной разработкой, а не “общаться с ботами”;
* проектов, где нужна дисциплина: DDD, clean architecture, ограничения по слоям, code style, политика миграций, security requirements.

То есть это не “помощник новичку написать hello world”.
Это скорее инструмент для **структурированной инженерной работы с ИИ в реальном проекте**.

# Как выглядит продукт в одном предложении

Это приложение, в котором пользователь ставит задачи ИИ-агентам в структурированном виде, система собирает правильный контекст, запускает исполнение в изолированной среде, валидирует результат и показывает его как набор инженерных артефактов для приемки.

# Основные сущности системы

Чтобы продукт был целостным, в нем должны быть понятные сущности.

## Проект

Проект это рабочее пространство, внутри которого живут:

* конфигурация,
* правила,
* шаблоны,
* карта кода,
* задачи,
* результаты запусков,
* агенты,
* маршрутизация.

У проекта есть своя идентичность.
Например один проект это backend на .NET с DDD, другой это React UI kit, третий это Go CLI.

## Guidelines

Это правила проекта.

Не просто markdown-файлы “на память”, а базовый нормативный слой системы.

Guidelines могут описывать:

* кодстайл,
* архитектурные принципы,
* naming rules,
* ограничения по слоям,
* работу с базой,
* политику тестирования,
* правила логирования,
* правила безопасности,
* миграции,
* формат PR summary,
* definition of done.

Пример:

* `backend-csharp-guidelines.md`
* `ddd-rules.md`
* `database-migrations-policy.md`
* `testing-policy.md`

## Prompt Templates / Task Templates

Это шаблоны постановки типовых задач.

Не надо каждый раз вручную писать структуру.
Пользователь выбирает шаблон, а система подставляет поля.

Например:

* добавить endpoint,
* реализовать repository,
* починить падающие тесты,
* провести рефакторинг,
* написать integration tests,
* обновить документацию,
* сделать migration.

## Task

Главная единица работы.

Task это формализованная задача, у которой есть:

* id,
* название,
* цель,
* scope,
* ограничения,
* контекст,
* выбранный агент,
* правила выполнения,
* критерии проверки,
* статус,
* результат.

## Context Pack

Это собранный пакет контекста для выполнения задачи.

Сюда входит:

* текст задачи,
* релевантные guidelines,
* нужные файлы,
* summaries,
* зависимости,
* связанные тесты,
* допустимые пути,
* запрещенные пути,
* validation rules.

## Agent

Это исполнитель.

В системе агент это не просто “LLM provider”, а логическая сущность со своей специализацией.

Например:

* Claude для сложных архитектурных правок,
* Codex для выполнения code tasks в изолированном окружении,
* Qwen для массовых низкорисковых операций,
* локальная модель для дешевых рутинных задач.

У агента могут быть свойства:

* специализация,
* сильные стороны,
* стоимость,
* скорость,
* ограничение контекста,
* режимы работы,
* доступные инструменты.

## Run

Run это конкретное выполнение задачи.

У одной задачи может быть несколько запусков:

* первый неудачный,
* второй после исправления,
* третий в другом агенте,
* четвертый с другим контекстом.

Run хранит:

* prompt snapshot,
* context snapshot,
* environment info,
* output,
* diff,
* logs,
* validation report,
* timestamps.

## Result / Artifact

Результат работы хранится как набор артефактов:

* patch,
* changed files,
* summary,
* logs,
* stdout/stderr,
* validation results,
* reviewer notes.

## Review Decision

Итог человеческой или автоматической приемки:

* accepted,
* rejected,
* needs_changes,
* reroute,
* retry.

# Полное описание функционала

Теперь по основным функциональным блокам.

# 1. Инициализация проекта

Система должна уметь создавать агентную среду внутри репозитория.

Например команда:

```bash
agentctl init
```

После этого в проекте создается структура вида:

```text
.agentctl/
  config.yaml
  agents.yaml
  routing.yaml
  guidelines/
  templates/
  tasks/
  runs/
  context/
  reviews/
```

### Для чего это нужно

Чтобы проект стал “понятным” для системы:

* где лежат правила,
* где задачи,
* как выбирать агентов,
* как выполнять задачи,
* как хранить артефакты.

### Почему это важно

Без проектной структуры система будет хаотичной.
Нужна единая точка, где живет вся мета-логика агентной разработки.

---

# 2. Система guidelines

Пользователь загружает или создает набор правил проекта.

Пример:

```bash
agentctl guidelines add ./docs/backend-guidelines.md
agentctl guidelines add ./docs/ddd-rules.md
agentctl guidelines add ./docs/testing-policy.md
```

Система индексирует эти правила и позволяет прикреплять их к задачам.

### Что могут содержать guidelines

Например guideline для backend может говорить:

* все бизнес-правила должны жить в Domain или Application;
* Infrastructure не должна протекать наружу;
* репозитории возвращают domain сущности;
* async везде, где есть I/O;
* публичные контракты нельзя ломать;
* все новые handler'ы покрываются unit tests;
* интеграционные тесты нужны, если меняется persistence.

### Для чего это сделано

Чтобы агент работал не “как умеет”, а по нормам конкретного проекта.

### Почему это важно

Одна и та же задача в разных проектах решается по-разному.
Нельзя надеяться, что агент сам угадает философию кода.

---

# 3. Шаблоны задач

Система должна поддерживать task templates.

Например есть шаблон “Add Repository”:

```yaml
name: add-repository
fields:
  - module
  - entity
  - database
defaults:
  validation:
    - dotnet build
    - dotnet test
attached_guidelines:
  - backend-guidelines
  - ddd-rules
```

Когда пользователь создает задачу по этому шаблону, система уже знает:

* какую структуру task spec сформировать,
* какие правила прикрепить,
* какие проверки предложить.

### Для чего это нужно

Чтобы сократить ручную работу и стандартизировать постановку задач.

### Почему это важно

Самая частая проблема не в плохом агенте, а в плохой задаче.
Шаблоны делают постановку задач более точной и повторяемой.

---

# 4. Создание задачи

Пользователь создает задачу, например:

```bash
agentctl task create
```

Или через параметры:

```bash
agentctl task create \
  --title "Implement refresh token repository" \
  --goal "Add repository for refresh tokens in auth module" \
  --agent claude \
  --allow "src/Auth/**" \
  --forbid "src/Billing/**" \
  --guide backend-guidelines \
  --guide ddd-rules
```

Система создает task spec примерно такого вида:

```yaml
id: TASK-001
title: Implement refresh token repository
goal: Add repository for refresh tokens in auth module
status: draft
agent: claude

scope:
  allowed_paths:
    - src/Auth/**
  forbidden_paths:
    - src/Billing/**
    - frontend/**

guidelines:
  - backend-guidelines
  - ddd-rules

context:
  include_files:
    - src/Auth/Application/Interfaces/IRefreshTokenRepository.cs
    - src/Auth/Infrastructure/DependencyInjection.cs
  include_patterns:
    - RefreshToken
    - TokenRepository

constraints:
  no_breaking_changes: true
  require_tests: true

validation:
  commands:
    - dotnet build
    - dotnet test
```

### Для чего это нужно

Чтобы каждая задача была четко формализована.

### Почему это важно

Если задача не формализована, агент неизбежно начинает додумывать за пользователя.

---

# 5. Scope и ограничения

Одна из важнейших фич системы это **жесткое ограничение области работы**.

В задаче должно быть указано:

* какие пути можно менять,
* какие нельзя,
* какие файлы обязательно прочитать,
* какие файлы считаются источником истины.

Пример:

```yaml
scope:
  allowed_paths:
    - src/Auth/Application/**
    - src/Auth/Infrastructure/**
  forbidden_paths:
    - src/Payments/**
    - deployment/**
must_read:
  - docs/backend-guidelines.md
  - src/Auth/Application/LoginHandler.cs
  - src/Auth/Infrastructure/DependencyInjection.cs
```

### Для чего это сделано

Чтобы агент не трогал лишнее и не расползался по проекту.

### Почему это важно

Большая часть “странных” правок агентами происходит именно потому, что им не очертили границы.

### Пример

Пользователь хочет изменить работу refresh token в auth-модуле.
Без ограничений агент может:

* залезть в user service,
* поменять DTO,
* переделать middleware,
* тронуть unrelated config.

С ограничениями он видит:

* работать можно только тут,
* источник истины вот эти файлы,
* за пределы области выходить нельзя.

---

# 6. Context Builder

Это один из самых важных модулей всей системы.

Когда задача готова, система должна собрать **контекстный пакет**.

Она не просто берет список файлов из task spec.
Она может автоматически добавить:

* связанные интерфейсы,
* ближайшие реализации,
* тесты рядом,
* summaries зависимых сущностей,
* куски документации,
* конфиги,
* недавние релевантные изменения.

### Как это работает

Например есть задача:

> реализовать persistence для refresh tokens

Context Builder может включить:

* интерфейс репозитория,
* текущий login handler,
* DI registration,
* entity mapping,
* соседние тесты,
* guideline про работу с persistence,
* summary слоя Application.

### Для чего это нужно

Чтобы агенту хватило контекста, но контекст оставался компактным.

### Почему это важно

Если контекста слишком мало, агент тупит.
Если контекста слишком много, агент тонет.

Context Builder это механизм баланса между этими крайностями.

---

# 7. Prompt Builder

После того как собран контекст, система строит итоговый execution prompt.

Он должен быть структурированным и не зависеть от пользовательского настроения.
То есть не “сделай красиво пожалуйста”, а четкая инструкция.

Пример итогового запроса:

```md
# Task
Implement refresh token repository in auth module.

## Goal
Add persistence for refresh tokens without breaking current auth flow.

## Allowed Paths
- src/Auth/Application/**
- src/Auth/Infrastructure/**

## Forbidden Paths
- src/Payments/**
- frontend/**

## Must Read
- docs/backend-guidelines.md
- src/Auth/Application/LoginHandler.cs
- src/Auth/Application/Interfaces/IRefreshTokenRepository.cs
- src/Auth/Infrastructure/DependencyInjection.cs

## Constraints
- Follow DDD boundaries
- Keep public API backward compatible
- Add tests for persistence logic
- Do not introduce breaking schema changes without migration

## Expected Output
- code changes
- summary of implementation
- list of modified files
- test results
- any risks or assumptions
```

### Для чего это сделано

Чтобы агент получал не абстрактный “промпт”, а нормализованный task contract.

### Почему это важно

Структурированная задача дает стабильность результата.

---

# 8. Routing между агентами

Система должна поддерживать несколько агентов и уметь выбирать, кому отправлять задачу.

Изначально это может быть ручной выбор:

* `--agent claude`
* `--agent codex`
* `--agent qwen`

Позже добавляется routing policy.

Пример:

```yaml
routing:
  - when: task.type == "architecture_refactor"
    agent: claude
  - when: task.type == "code_generation"
    agent: codex
  - when: task.type == "bulk_tests"
    agent: qwen
```

### Для чего это нужно

Чтобы использовать сильные стороны разных моделей.

### Почему это важно

Одна модель хороша в одном, другая в другом.
Нет смысла все задачи слать одному исполнителю.

### Пример

Задача “прочитать большую кодовую базу и предложить рефакторинг” лучше уйдет в более сильного аналитического агента.
Задача “сгенерировать пачку тестов по понятному шаблону” может уйти в более дешевую модель.

---

# 9. Изолированное выполнение

Каждая задача должна выполняться в своей изолированной среде.

Например:

* git worktree,
* временная папка,
* отдельная ветка,
* контейнер.

### Для чего это сделано

Чтобы задачи не мешали друг другу и могли исполняться параллельно.

### Почему это важно

Если несколько агентов работают по одному и тому же рабочему дереву, начнется катастрофа:

* конфликтующие изменения,
* перезапись файлов,
* потеря контекста,
* невозможность понять, кто что сделал.

### Пример

Есть три задачи:

* исправить auth module,
* написать тесты для billing,
* обновить документацию API.

Все три запускаются одновременно.
Каждая задача получает отдельный worktree.
После завершения система показывает результаты по отдельности.

---

# 10. Асинхронная очередь задач

Это центральная фича системы.

Пользователь не обязан ждать завершения текущей задачи, чтобы поставить следующую.
Он может создать целую очередь задач.

Например:

```bash
agentctl task run TASK-001
agentctl task run TASK-002
agentctl task run TASK-003
```

Система сама ведет статусы:

* draft,
* queued,
* preparing_context,
* running,
* validating,
* review,
* completed,
* failed,
* rejected.

### Для чего это нужно

Чтобы работа с агентами была похожа на реальную инженерную систему, а не на последовательную переписку.

### Почему это важно

Главное преимущество ИИ-исполнителей в том, что их можно распараллелить.
Если каждый раз сидеть и ждать ответа в одном чате, смысл частично теряется.

### Пример

Ты ставишь три задачи трем агентам, идешь смотреть результаты позже.
Это уже похоже не на “чат с ботом”, а на “диспетчеризацию цифровых сотрудников”.

---

# 11. Validation Pipeline

После выполнения задачи система должна прогонять проверки.

Это могут быть:

* build,
* lint,
* format,
* unit tests,
* integration tests,
* security checks,
* custom policy checks.

Пример:

```yaml
validation:
  commands:
    - dotnet build
    - dotnet test tests/Auth.UnitTests
    - dotnet test tests/Auth.IntegrationTests
```

### Для чего это сделано

Чтобы принимать не “на словах”, а по проверяемым критериям.

### Почему это важно

Агент может красиво объяснить решение, но код все равно может не собираться.

### Пример

Задача выполнена, но validation показывает:

* build passed,
* 14 unit tests passed,
* 2 integration tests failed.

Пользователь сразу видит, что задача формально не завершена.

---

# 12. Результаты как артефакты

После выполнения система сохраняет результат в виде файлов.

Например:

```text
.agentctl/
  runs/
    TASK-001/
      prompt.md
      context.md
      summary.md
      diff.patch
      changed_files.json
      logs.txt
      validation.json
      metadata.json
```

### Для чего это нужно

Чтобы результат был наблюдаемым, проверяемым и воспроизводимым.

### Почему это важно

Чат-ответ это эфемерная штука.
Артефакт можно:

* сравнить,
* сохранить,
* проверить,
* использовать для аудита,
* показать другому человеку,
* проанализировать позже.

---

# 13. Просмотр diff и итогов

Система должна уметь показывать:

* список измененных файлов,
* patch,
* сводку по изменениям,
* лог выполнения,
* результаты validation,
* риски,
* предположения агента.

### Для чего это нужно

Чтобы пользователь принимал решение на основе технических данных.

### Пример

Команда:

```bash
agentctl result show TASK-001
```

Показывает:

* агент: claude
* изменено файлов: 4
* пройдено тестов: 16/18
* статус: review
* summary: реализован repository, добавлен registration в DI, обновлены tests
* risk: миграция БД не добавлена

Это намного полезнее, чем просто читать длинный ответ в чате.

---

# 14. Review и приемка

После того как задача выполнена и проверки завершены, пользователь должен принять решение.

Команды:

```bash
agentctl task accept TASK-001
agentctl task reject TASK-001 --reason "Touched forbidden paths"
agentctl task rerun TASK-001
agentctl task route TASK-001 --agent codex
```

### Для чего это нужно

Чтобы человек оставался финальным контролером качества.

### Почему это важно

ИИ-агенты это исполнители, но не финальные владельцы проекта.

### Пример

Если агент изменил не ту часть проекта, задача отклоняется.
Если все корректно, задача принимается и может быть смержена или сохранена как завершенная.

---

# 15. Второй агент как reviewer

Очень сильная фича: разделить роли.

Один агент делает реализацию, другой делает review.

Например:

* Claude пишет код,
* Codex или другой агент проверяет diff,
* человек смотрит уже агрегированный итог.

### Для чего это нужно

Чтобы снизить риск ошибок и получить независимую проверку.

### Почему это важно

Один и тот же агент, который написал код, часто будет слишком “лоялен” к собственному результату.

### Пример

Executor говорит:

> задача выполнена, добавлен репозиторий, тесты добавлены.

Reviewer говорит:

> код рабочий, но нарушен guideline: Application слой теперь зависит от Infrastructure-специфики.

Это уже очень полезный сигнал.

---

# 16. История запусков

У каждой задачи должна быть история:

* первый запуск,
* повторный запуск,
* запуск в другом агенте,
* изменения конфигурации,
* смена контекста.

### Для чего это нужно

Чтобы можно было анализировать, почему задача сработала или не сработала.

### Почему это важно

В инженерной системе важна не только текущая версия, но и эволюция решения.

---

# 17. Метрики по агентам

Со временем система может собирать статистику:

* какой агент лучше справляется с каким типом задач,
* среднее число retries,
* среднее число touched forbidden paths,
* среднее качество по validation,
* среднее время выполнения,
* цена выполнения.

### Для чего это нужно

Чтобы routing стал умнее.

### Почему это важно

Не нужно полагаться на интуицию, когда можно опираться на реальные данные.

---

# 18. Режимы работы

Система может поддерживать несколько режимов.

## Строгий режим

Максимум ограничений:

* жесткий scope,
* запрет на лишние файлы,
* обязательный review,
* обязательный validation.

Подходит для продовых задач.

## Быстрый режим

Меньше ограничений, быстрый запуск, меньше проверок.

Подходит для черновой генерации и прототипов.

## Исследовательский режим

Агент может не менять код, а только анализировать, предлагать решения, строить план.

Подходит для архитектурных исследований.

# Как это будет работать на практике

Теперь соберу несколько живых сценариев.

# Сценарий 1. Простая локальная задача

Ты работаешь над backend на .NET и хочешь добавить репозиторий refresh token.

Создаешь задачу:

```bash
agentctl task create \
  --title "Implement refresh token repository" \
  --goal "Add persistence for refresh tokens in auth module" \
  --agent claude \
  --allow "src/Auth/**" \
  --guide backend-guidelines \
  --guide ddd-rules
```

Дальше:

```bash
agentctl task run TASK-001
```

Система:

1. собирает guidelines;
2. находит интерфейсы и DI registration;
3. добавляет соседние тесты;
4. формирует context pack;
5. создает worktree;
6. запускает агента;
7. сохраняет diff;
8. гоняет `dotnet build` и `dotnet test`.

Потом ты смотришь:

```bash
agentctl result show TASK-001
agentctl result diff TASK-001
```

И видишь:

* изменено 4 файла,
* тесты прошли,
* summary есть,
* forbidden paths не затронуты.

Принимаешь:

```bash
agentctl task accept TASK-001
```

---

# Сценарий 2. Параллельная работа нескольких агентов

У тебя три независимые задачи:

* написать тесты,
* исправить auth module,
* обновить docs.

Ты запускаешь все три.

Пока они идут, ты не сидишь в чате и не ждешь.
Система сама ведет состояния.

Через какое-то время делаешь:

```bash
agentctl task list
```

И видишь:

* TASK-001: review
* TASK-002: running
* TASK-003: completed

Открываешь только те задачи, которые уже готовы к приемке.

Это и есть главное отличие от чатового сценария: ты работаешь не с диалогами, а с очередью инженерных операций.

---

# Сценарий 3. Один агент пишет, другой проверяет

Ты запускаешь задачу так, чтобы после исполнения включился reviewer.

```bash
agentctl task run TASK-004 --reviewer codex
```

После выполнения основного агента система автоматически:

* берет diff,
* summary,
* validation report,
* отправляет все reviewer-агенту.

Reviewer возвращает:

* замечания,
* список рисков,
* возможные нарушения guidelines.

Ты уже принимаешь решение, имея две точки зрения:

* исполнителя,
* проверяющего.

---

# Сценарий 4. Большая задача с декомпозицией

Позже система может поддерживать planner mode.

Ты ставишь большую задачу:

> Реализовать полноценную ротацию refresh token с защитой от reuse, логированием и тестами.

Planner разбивает это на подзадачи:

* TASK-101: persistence layer
* TASK-102: application logic
* TASK-103: integration tests
* TASK-104: docs update

Дальше каждая задача уходит в отдельный pipeline.
Это уже почти маленькая агентная фабрика.

# Почему именно CLI как первый M4P

Потому что тебе сейчас нужно проверить механику, а не рисовать красивую обертку.

CLI идеально подходит для проверки:

* структуры task spec,
* logic of context packing,
* isolated runs,
* validations,
* artifacts,
* acceptance flow.

CLI естественно дружит с:

* git,
* worktree,
* bash-командами,
* файлами конфигов,
* automation,
* CI/CD.

То есть первый MVP как консольная утилита это очень сильное и логичное решение.

# В чем главная сила идеи

Сила этой идеи в том, что она смещает точку контроля от “разговора” к “инженерному процессу”.

Неважно, насколько хороши модели.
Если у тебя нет:

* нормальной постановки задачи,
* хорошего контекста,
* ограничений,
* артефактов результата,
* проверки,
* приемки,

то ты получаешь хаос с красивыми ответами.

А здесь ты строишь именно **систему управления работой агентов**.

Это уже не “спросить ИИ что-то про код”.
Это:

* выдать задачу,
* обеспечить контекст,
* проконтролировать границы,
* валидировать результат,
* принять решение.

Именно поэтому идея выглядит сильной.

# Если совсем сжать суть

Продукт должен быть таким:

**Консольная утилита для агентной разработки, в которой задачи формализуются, контекст собирается автоматически, агенты выбираются осознанно, выполнение происходит в изоляции, результат сохраняется как набор артефактов, а человек принимает итог через review-процесс.**

Не чат.
Не copilot.
Не игрушка поверх LLM.

А **диспетчерская система управляемой разработки с ИИ-исполнителями**.
